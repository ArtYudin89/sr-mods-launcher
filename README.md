# SR Mods Launcher

Лёгкий лаунчер для Space Rangers HD: ведёт список модов, скачивает и ставит их в папку
`Mods`, показывает даты обновления/установки, запускает игру. Сборки (профили), светлая/тёмная тема.

> 🎮 **Новичку:** пошаговое руководство для игрока — в [ИНСТРУКЦИЯ.md](ИНСТРУКЦИЯ.md).
> Этот README — техническая справка (сборка, форки, архитектура).

## Интерфейс (новый, на pywebview)

Актуальный интерфейс — **веб-технологии в нативном окне** (pywebview + Edge WebView2), исходники
в каталоге **`webui/`**: `app.py` (мост JS↔Python поверх `launcher_core`) + `web/`
(`index.html`/`style.css`/`app.js`). Движок (`launcher_core.py`) общий со старым GUI.

- **Светлая и тёмная темы** — переключатель 🌙/☀ в шапке, выбор запоминается.
- **Заточено под новичков:** онбординг «Шаг 1 — выберите папку игры», подсказки простым языком,
  подтверждения перед установкой/записью/удалением, технические настройки спрятаны под «Дополнительно».
- **Легенда состояний** (кнопка `ⓘ Обозначения`) и **фильтр** (`⚲ Фильтр`): любой набор состояний
  (модель «показывать только отмеченные»), плюс «только подключённые в игре» / «только в сборке».
- **Колонка раздел/папка** показывает измерение, обратное группировке (группируешь по папкам —
  колонка «Раздел», и наоборот).

Старый интерфейс на **tkinter** (`launcher.py`) оставлен как legacy и работает по-прежнему, но
развитие идёт в `webui/`. Требование для нового UI — **Edge WebView2 Runtime** (штатно есть на
Windows 10/11).

## Что нового в v0.16.0

- **Режим обучения (тур по интерфейсу).** При первом запуске автоматически показывается
  пошаговый тур с подсветкой элементов (папка игры → сборка → добавить мод → список →
  установить → проверить/обновить → база → играть). Листается стрелками ←/→, закрывается
  Esc. Повторить в любой момент — кнопка **🎓** в шапке. Флаг «пройдено» хранится в конфиге.
- **Кнопка справки ❔.** Открывает окно с краткой справкой (7 пунктов) и ссылкой
  **«🌐 Полная справка на GitHub»** (ведёт на [ИНСТРУКЦИЯ-ПРОСТАЯ.md](ИНСТРУКЦИЯ-ПРОСТАЯ.md)),
  а также кнопкой «🎓 Пройти обучение».
- **Исправлена «вечная переустановка лагеря».** Установка «весь лагерь» больше не
  переписывает одни и те же файлы на каждом заходе. Раньше юниты (`base`/`fixes`/разные
  моды), делящие один целевой файл с разным содержимым, затирали друг друга — сверка
  хешей никогда не сходилась к нулю, и лаунчер каждый раз «ставил» ~200 файлов. Теперь
  лагерь собирается **одним проходом** (`reconstruct_camp`): манифесты всех юнитов
  сливаются в единый набор с приоритетом (кто позже в порядке установки — тот побеждает
  спорный файл), после чего идёт одна сверка и одна запись. Повторная установка того же
  лагеря проходит за секунды («скачивать не нужно»). Финальное состояние игры прежнее.

## Что нового в v0.15.4

Доступность (без изменений поведения для мышиной аудитории):

- **Клавиатурная навигация дерева модов.** Стрелки ↑/↓/Home/End перемещают курсор; →/← раскрывают/сворачивают группу и переходят к ребёнку/родителю; Space — выделить строку, Enter — в группе свернуть/раскрыть, в строке мода — переключить «в сборке»; `i` — окно с описанием; ContextMenu/Shift+F10 — контекстное меню. Курсор выживает при обновлении дерева (roving tabindex, флаг `_treeRefocus`). ARIA: `role=tree/treeitem`, `aria-level`, `aria-expanded`, `aria-selected`.
- **Focus-trap в модальных окнах.** `show()` запоминает прежний фокус и ставит Tab-цикл внутри `.modal`; `hide()` снимает его и возвращает фокус; поддерживается стек вложенных окон (`_trap`). Закрытие по фону — через `hide()`.
- **Видимая рамка фокуса (`:focus-visible`).** Рамка видна только при навигации с клавиатуры; при работе мышью — без изменений.
- **WCAG AA-контраст.** Аудит приглушённого текста (`--muted`) прошёл AA во всех токенах (тёмная/светлая). Единственный провал — метки времени в журнале (`.l-ts`, `opacity:.55` давало 2.3:1) → opacity убран, теперь 5.0:1.

## Возможности (v0.12 — новый интерфейс)

- **Профиль = переносимый набор.** Профиль хранит постоянно: **набор подключённых модов** (как
  `ModCFG`), **путь к игре** (и задел на гит-форк агрегатора). Такой профиль можно отдать другому
  игроку. **База определяется автоматически** по содержимому `Mods` (маркеры redux/universe/original)
  и показана над списком — это **факт диска**, а не часть профиля. Поле «База» в настройках ⚙ →
  «Дополнительно» **необязательно** — только пометка для переносимой сборки без установленной игры,
  на установку не влияет.
- **Подключение модов (ModCFG) ⟷ профиль.** Две новые колонки дерева: **«Подключен в игре»** (по
  `Mods/ModCFG.txt` — что игра реально грузит) и **«Подключен в профиле»** (набор `enabled` профиля).
  Подключение в сборке переключается **кликом по галочке** в колонке; при этом считается **каскад**
  (`Dependence`/`Conflict`, матч по `Name` из ModuleInfo, не по папке) и при необходимости
  показывается подтверждение: при включении — что **➕ подключится** (зависимости) и **➖ отключится**
  (конфликтующие + транзитивно зависящие от них), при выключении — какие включённые моды **зависят**
  от снимаемого и отключатся следом; если обязательной зависимости нет нигде — подключение
  **блокируется** (`plan_enable`/`plan_disable` в `launcher_core`). Панель
  **«Подключение в игре»**: **⟸ Считать из игры в сборку** (ModCFG → сборка) и **⟹ Записать сборку в
  игру** (сборка → `ModCFG.txt`, предупреждает о модах не на диске). Прочие строки и кодировка
  ModCFG сохраняются.
- **Дерево Лагерь → Пак → Мод.** В одном списке и установленное на диске (колонка «Установлен» с
  датой), и добавленное в набор (`➕ добавлен`). Моды **набора профиля**, которых ещё нет на диске,
  показываются строками `📥 доступен` / `⚠ недоступен` (по каталогу); ПКМ «⬇ Добавить в набор»
  ставит их в очередь скачивания. Переключатель группировки **«По папкам / По разделам»**
  (`Section` из ModuleInfo) и переключатель имени **«Имя: папка / как в игре»** (поле `Name` из
  ModuleInfo — у Pol*/Shu*-модов папка и игровое имя различаются; рядом серым показано имя папки).
  Поиск ищет и по папке, и по `Name`. Фильтр **«Только требующие
  внимания»** скрывает `✅ установлен`. ПКМ по строке — открыть папку мода / обновить с сохранением
  правок. Кнопки развернуть/свернуть. Дисковые и профильные моды группируются единообразно по папке/разделу.
  В шапке — **бегущий итог очереди** «в наборе: N · ~X ГБ».
- **Очередь набора — сессионная.** Добавленное через ＋ — это очередь на установку; исчезает при
  ручном удалении, после установки (мод становится дисковым) или при перезапуске. **`－` убирает
  несколько выделенных** и **группу целиком** (лагерь/пак разворачивается в записи набора).
- **Раскладка установки = как в игре.** `install_route`/`install_target`: контент `Mods/` → в `Mods`,
  файлы базового пака (`Rangers.exe`, `matrix/`, `CFG/`, `DATA/`) → в **корень игры**; staging-обёртки
  (`*_unpacked/`, `.temp/`), Inno-плейсхолдеры (`{userdocs}`) и мусор инсталлятора (`.iss`,
  `.exe`-установщики, readme) — **пропускаются**. Полная установка лагеря идёт по порядку
  **база → паки/моды → фиксы паков → фиксы базы**.
- **Параллельная загрузка.** Части качаются в несколько потоков (общая `requests.Session` с пулом) —
  на канале с большим RTT/DPI один поток не насыщает полосу. Размер потоков — env `SRML_PARALLEL`
  (по умолч. 6).
- **Реальные размеры.** Перед крупной установкой — реальный объём (из `state/packs.json`,
  `code/asset/bytes` на пак) с пометкой про базовый пак.
- **Сохранение правок при обновлении (3-way merge).** Кнопка **`🔀 Обновить (сохранить правки)`**:
  если игрок руками менял файлы мода, при обновлении они **по возможности сохраняются**. Текст
  сливается построчно (diff3, чистый Python — без git); непересекающиеся правки — автоматически,
  настоящие конфликты — **с предпросмотром и выбором по каждому файлу** (оставить мой / взять новый /
  сохранить оба как `.srnew`). Работает и для модов набора, и для **дисковых** (из «Установлено в игре»).
  База 3-way не хранится байтами — снимок установки (`relpath→sha256`) лежит вне `Mods` в
  `~/.sr-mods-launcher/snapshots/`, содержимое подтягивается по sha. Можно выбрать несколько модов —
  обновятся по очереди.
- **Индексация модов на диске** (кнопка **`🗂`**, при первом запуске — предложение): локально, без
  скачивания, определяет базу и помечает моды `✅ знаком каталогу` / `❓ не в каталоге`. Инкрементально
  (sha файла переиспользуется по size+mtime) — быстро при повторе, учитывает добавленные/удалённые моды.
  **После установки лаунчером индекс пополняется сразу** — sha берётся из скачанных байт (`sha_sink`/
  `sha_cache`), без повторного чтения с диска; реиндекс не идёт «с нуля».
- **Установка с зависимостями** (`🧩`): **резолвит `Dependence`** из ModuleInfo (замыкание зависимостей),
  показывает конфликты (`Conflict`) и список к установке с подтверждением. «Установить весь набор`
  пропускает уже актуальные (идемпотентность). Код и ассеты — content-addressed части с Hugging Face
  (публично, без токена).
- **Определение установленной базы** по уникальным маркерам в `Mods/` (redux/original/universe).
- **Совместимость** (`🛡`): паки набора (одна база, фикс→родитель, сейвы) **+ проверка набора «в
  сборке» по двум направлениям** (`check_enabled_compat`, матч по `Name`): (1) у включённого мода
  есть зависимость не в сборке (с пометкой, доступна ли она в каталоге/на диске или отсутствует
  вовсе); (2) два конфликтующих мода оба включены. Только показывает.
- **Прогресс и отмена:** индикатор `пак K/T · часть N/M · ↓ X.XX ГБ` — агрегатные гигабайты растут
  непрерывно (видно, что не зависло даже на крупной части). Кнопка **`✖ Отмена`** прерывает операцию;
  на время операции интерфейс блокируется. Чекбокс **«Подробный лог»** — краткий/детальный вывод.
  Сетевые запросы — с авто-повтором (устойчивость к обрывам).
- **Проверка обновлений пофайловой сверкой** (`⟳`): хеши файлов на диске (из индекса, инкрементально)
  сверяются с **опубликованными манифестами** (с учётом форков). Мод подсвечивается **`⬆ обновление`**,
  если у опубликованной версии есть файл с другим sha или отсутствующий на диске. Надёжно — сверка
  **по каждому файлу** (а не по хэшу всего набора, который давал ложные срабатывания). При обновлении с
  сохранением правок в диалоге есть галка **«Запомнить решения для остальных модов»** — выбор по типу
  конфликта (текст/бинарь/удалён) применяется к остальным модам пакета без повторных вопросов.
- **Дополнительные репозитории (форки).** В Настройки ⚙ → «Дополнительно» можно добавить **список**
  доп. репозиториев-агрегаторов (с приоритетом ↑↓). Они **накладываются поверх основного по файлам**:
  мод обязан быть в основном репозитории (форк не вводит новые моды), а его файлы форк может
  **заменить или дополнить** — выигрывает самый приоритетный репозиторий, где есть файл. Каждый файл
  качается из своего хранилища (HF) автоматически. Наложение работает во всех путях: отдельный мод,
  набор с зависимостями, обновление-с-правками и установка целого лагеря/базового пака.
- **Горячие клавиши** Ctrl+C/V/X/A работают в полях ввода при любой раскладке.
- Профили (переносимые наборы: база + подключённые моды + путь), запуск игры, выбор папки игры с
  автопоиском Steam.

Запись очереди установки бывает трёх типов: `unit` (пак или один мод агрегатора), `camp` (весь
лагерь), `desc` (мод по дескриптору/форк-URL). Профиль дополнительно хранит постоянные поля:
`base` (выбранная база), `enabled` (подключённые моды, как ModCFG), `game_path`, `fork`. Очередь
установки (`mods`) — сессионная (см. ИНСТРУКЦИЯ.md), `enabled`/`base` — постоянные и переносимые.

**Проверка совместимости** (кнопка `🛡 Проверить совместимость`): по `state/packs.json`
агрегатора лаунчер проверяет набор — ровно одна **база** (пак с `Rangers.exe`), наличие
родительских паков для **фиксов** (`fix_parent`), несовместимость сейвов при смене базы,
обязательность base/fix-обновлений; плюс конфликты/зависимости модов. Только показывает.

**Форки (дополнительные репозитории).** В Настройки ⚙ → «Дополнительно» задаётся список доп.
репозиториев-агрегаторов (той же структуры: `descriptors/catalog.json` + манифесты + свой
`asset_index`/HF), в порядке приоритета. Лаунчер **накладывает их поверх основного пофайлово**: мод
ставится, только если он есть в основном каталоге; его файлы форк может заменить или дополнить, причём
для каждого файла выигрывает самый приоритетный репозиторий, где этот файл есть, и качается из его
хранилища. Так форк = «патч/переопределение» поверх базовой раздачи. (Старое поле «URL форка» для
одного мода-дескриптора оставлено, но список репозиториев покрывает тот же сценарий.)

## Запуск из исходников (dev-режим)

Новый интерфейс:

```
pip install -r requirements.txt pywebview
python webui/app.py            # --debug откроет инструменты разработчика
```

Зависимости: `requests` + `pywebview` (на Windows тянет `pythonnet`/WebView2). Старый
tkinter-вариант по-прежнему: `python launcher.py` (нужен только `requests`, tkinter в составе Python).

## Тестирование

В `webui/` лежат два тестовых файла, не требующих сети и GUI:

| Файл | Уровень | Что покрывает |
|------|---------|---------------|
| `test_ui_logic.py` | L1 — логика | Состояния профиля, `_needs_order`, `plan_actionable_sha`, чистая бизнес-логика (без диска) |
| `test_integration.py` | L3 — интеграция | Реальный диск + мокнутая сеть: установка, detect-update, merge, cancel, ModCFG, get_tree, E2E-цепочки |

```
# из корня репозитория
python webui/test_ui_logic.py       # 26 проверок
python webui/test_integration.py    # 180 проверок
```

Оба запускаются напрямую (без pytest), пишут `[OK]`/`[FAIL]` в stdout,
возвращают exit-code 1 при провалах.

### Архитектура test_integration.py

- **`Api.__new__`** — создаёт экземпляр в обход `__init__` (нет сети, нет pywebview).
- **`TempGame`** — временная папка `game/Mods/` + `profiles/`; убирается после группы.
- **`patch.object(core, 'reconstruct_unit', ...)`** — мок сетевых вызовов. Патч живёт через `.start()/.stop()`, а не `with`-блоком, чтобы покрывать фоновый поток.
- **`wait_idle(api)`** — опрашивает `api.busy` до False (≤ 8 с) для синхронизации с daemon-потоком.

## Сборка .exe (Windows)

Требуется Python 3.8+, PyInstaller, pywebview. Из **корня репозитория**:

```
cd webui
python -m PyInstaller --clean --noconfirm SRModsLauncher.spec
```

Результат: `webui\dist\SRModsLauncher.exe` — самодостаточный файл (~11 МБ; Python, движок и фронт
внутри). Spec кладёт `web/` в бандл (`_MEIPASS/web`), а конфиг/`profiles`/`Mods`-фолбэк создаются
рядом с .exe. Требует **Edge WebView2 Runtime** (штатно на Win10/11).

Замечания:
- `console=False` — без консольного окна.
- **Не храните ссылку на окно как атрибут js_api-объекта** — pywebview сериализует его и уходит в
  бесконечную рекурсию (`…AccessibilityObject…`), окно виснет. Держим окно в модульной глобальной.
- Антивирус: `--onefile` иногда помечается ложно. В Releases при необходимости кладём и
  `SRModsLauncher-source.zip` (исходники + bat) на случай, если exe блокируется.

### RWT-сборка (release with tests) — больше не нужна

Репозиторий агрегатора теперь **публичный**, поэтому обычный exe работает у всех **без токена**.
RWT-механизм (вшитый токен через `embedded_secrets.py`) оставлен для совместимости/приватных форков,
но для штатной раздачи не требуется. ⚠ Перед публичной сборкой убедись, что `embedded_secrets.py`
**отсутствует** (иначе токен вшивается из-за `hiddenimports`).

## Репозиторий агрегатора — публичный

Основной репозиторий модов **публичный**: лаунчер читает каталог/манифесты с
`raw.githubusercontent.com` **без токена и без лимита 60 запросов/час** (Contents API используется
только при заданном токене — для приватных репозиториев/форков). Ассеты и код — content-addressed части
с Hugging Face (тоже публично). Токен в настройках нужен лишь для приватного репозитория/форка.

## Темы

Светлая/тёмная тема переключаются прямо в интерфейсе (🌙/☀), выбор хранится в
`launcher_config.json` (`theme`). Палитры заданы CSS-переменными в `webui/web/style.css`
(`:root[data-theme="dark"]` / `[data-theme="light"]`).
